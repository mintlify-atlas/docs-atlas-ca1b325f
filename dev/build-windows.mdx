title: "Building DOSBox Staging on Windows: A Developer Guide"
description: "Compile DOSBox Staging on Windows using Visual Studio 2022 with the Clang/LLVM toolchain and vcpkg to install and manage all project dependencies."
---

The primary and fully supported Windows toolchain is Visual Studio 2022 with the Clang/LLVM compiler and vcpkg for dependency management. This is the toolchain the team uses to produce official release builds. A Meson-based build using MSYS2 is also possible but is deprecated and will likely be removed in a future release.

<Note>
  CMake support on Windows is currently an experimental, internal work in progress. The CMake import workflow described below requires Visual Studio 2022 17.x or later.
</Note>

---

## Build using Visual Studio (solution file)

This is the quickest path to a working build and uses the traditional Visual Studio solution file.

<Steps>
  <Step title="Install Visual Studio Community 2022">
    Download and run the installer from [https://visualstudio.microsoft.com/vs/community/](https://visualstudio.microsoft.com/vs/community/).

    In the installer, make sure to select **C++ Clang tools for Windows** under the **Desktop development with C++** workload.
  </Step>
  <Step title="Set up vcpkg">
    Recent versions of Visual Studio ship with a bundled copy of vcpkg that only needs to be initialised. Open a **Visual Studio Developer Command Prompt** and run:

    ```shell
    vcpkg integrate install
    ```

    This configures MSVC to automatically find, build, and link all vcpkg-managed dependencies.

    <Note>
      A standalone installation of vcpkg also works. See the [vcpkg quick-start guide](https://github.com/Microsoft/vcpkg#quick-start-windows) for details.
    </Note>
  </Step>
  <Step title="Open and build the solution">
    Start Visual Studio and open `vs\dosbox.sln`. Set the solution platform to **x64**, then press **Ctrl+Shift+B** to build all projects.

    <Tip>
      The first build for each configuration will compile all vcpkg dependencies automatically. This can take a significant amount of time — subsequent builds are much faster.
    </Tip>
  </Step>
</Steps>

## Build using Visual Studio with CMake

This workflow imports the repository as a CMake project and gives you access to all CMake presets directly from the Visual Studio toolbar.

### Prerequisites

- Python 3 with pip — only required if you want to build the [offline documentation](#offline-documentation). Download the installer from [https://www.python.org/downloads/](https://www.python.org/downloads/) and check **"Add Python to PATH"** during installation.

### Import and build

<Steps>
  <Step title="Remove any existing DOSBox Staging projects from Visual Studio">
    Close any open DOSBox Staging project or solution. Exit Visual Studio, then delete the `.vs` folder from your local repository if it exists.
  </Step>
  <Step title="Open the repository folder in Visual Studio">
    Start Visual Studio and select **Open a local folder** from the startup screen, or use **File > Open > Folder** from the menu. Navigate to your local DOSBox Staging repository.
  </Step>
  <Step title="Wait for CMake project setup">
    Visual Studio will display a CMake splash screen while it configures the project. Once finished, the available CMake configurations appear in the second dropdown below the menu bar, next to **Local Machine**.
  </Step>
  <Step title="Select a configuration and build">
    Choose either **debug-windows** or **release-windows** from the configuration dropdown, then select **Build > Build All** or press **F7**.

    <Note>
      You may need to build twice the first time, and again after modifying `CMakeLists.txt` files.
    </Note>
  </Step>
</Steps>

## Offline documentation

Self-contained HTML documentation can optionally be built alongside the main project. The output is placed at `build\<preset>\Resources\docs\` and matches the documentation bundled with release packages.

Documentation building is **off by default**. Use one of the dedicated manual presets to enable it:

| Preset | Description |
|---|---|
| `debug-windows-manual` | Debug build with documentation |
| `release-windows-manual` | Release build with documentation |

These presets appear in Visual Studio's configuration dropdown. From the command line:

```shell
cmake --preset release-windows-manual
cmake --build --preset release-windows-manual
```

To rebuild only the documentation after editing content:

```shell
cmake --build --preset release-windows-manual --target rebuild_documentation
```

Alternatively, enable documentation with any preset by passing the flag explicitly:

```shell
cmake --preset release-windows -DOPT_DOCUMENTATION=ON
```

### Prerequisites

Python 3 with pip must be installed and available on the system `PATH`. Download the installer from [https://www.python.org/downloads/](https://www.python.org/downloads/) and ensure **"Add Python to PATH"** is checked. The default installation includes pip — do not uncheck it.

<Warning>
  The Python installation included with Visual Studio's "Python development" workload may not be on the system `PATH`. A standalone Python installation is recommended.
</Warning>

### Rebuilding after content changes

Changes to files under `website\docs\` do not trigger an automatic rebuild. To rebuild documentation after editing:

<CodeGroup>
  ```shell using the rebuild target
  cmake --build --preset release-windows --target rebuild_documentation
  ```

  ```shell invalidating the build stamp (Git Bash or WSL)
  touch website/mkdocs.yml
  ```
</CodeGroup>

To force a complete documentation rebuild from scratch:

```shell
del build\release-windows\_mkdocs_build_stamp
```

To remove all MkDocs caches from the source tree:

```shell
cmake --build --preset release-windows --target clean-manual
```

<Note>
  If Python is unavailable or missing required modules, the build proceeds without documentation. A warning is shown during CMake configuration, but the build is never aborted.
</Note>

## Troubleshooting

### Random build failures

Try **Build > Clean All**, then **Project > Rescan Solution**. In most cases, **Build > Build All** succeeds after this.

If errors persist or those menu items are unavailable, delete the DOSBox Staging project in Visual Studio, delete the `.vs` folder inside your local repository, then follow the [CMake import steps](#import-and-build) above to reimport the project.

### CMake configurations have disappeared from the toolbar

Use **Project > Rescan Solution**. If the configurations do not reappear, delete the project from Visual Studio, delete the `.vs` folder, then reimport the project.
